# PolyMind 产品规划

## 产品定位

PolyMind 是一个面向复杂思考任务的多 AI 智能体讨论空间。它帮助用户把一个问题交给多个具备不同视角、模型能力和角色设定的 AI 参与者，让它们进行结构化讨论，并最终形成可以行动的结论。

产品不应呈现为「会议室」或「视频会议替代品」。更准确的体验隐喻是：

- 一个可以组织多种思维视角的 AI 思考画布。
- 一个能让专家型智能体相互补充、质疑和收敛的讨论空间。
- 一个面向项目、代码、论文、商业、学习和写作的多模型推理工作台。

核心价值：

- 从单 AI 回答升级为多视角讨论。
- 从一次性生成升级为可观察、可控制、可总结的思考过程。
- 从固定人设升级为可配置的 AI 参与者编排。

## 目标用户

- 产品经理、创业者：进行需求拆解、商业分析、路线图讨论。
- 工程师、技术负责人：进行架构方案评审、代码方案推演、风险识别。
- 学生、研究者：进行论文选题、文献观点整理、学习辅导。
- 写作者、设计师：进行选题发散、改稿建议、风格对照。
- 个人知识工作者：用多个 AI 视角帮助自己做决策和复盘。

## MVP 范围

第一版必须打通以下主路径：

1. 用户进入首页或空间列表页。
2. 用户创建一个讨论空间，输入主题、目标和上下文。
3. 用户添加 1 到 6 个 AI 参与者。
4. 每个 AI 参与者可以选择 Provider、模型和调用配置。
5. 每个 AI 参与者可以配置角色、人设、背景、发言风格和任务定位。
6. 用户也可以跳过角色配置，使用默认通用助手。
7. 用户启动讨论，多个 AI 按调度器依次流式发言。
8. 用户可以查看讨论过程、当前发言者、轮次和状态。
9. 用户可以暂停、继续或结束讨论。
10. 用户生成总结，系统输出共识、分歧、建议和后续问题。
11. 空间、参与者、消息和总结保存到数据库。

MVP 体验重点：

- 创建流程要轻，不要一开始让用户填大量表单。
- 讨论过程要有明显的「多智能体感」，不能只是单个聊天窗口换名字。
- 总结要结构化，不能只是把聊天记录压缩成一段话。
- UI 必须精致、现代、响应快，有空间层次和柔和动效。

## 不在第一版实现的功能

- 语音通话、视频会议、屏幕共享。
- 多人实时协作、团队空间、权限系统。
- 付费订阅、额度系统、发票、组织账单。
- 公开智能体市场、模板市场。
- 文件知识库、RAG、长期记忆。
- 复杂工作流编排器、可视化节点图。
- 浏览器自动操作、代码执行沙箱。
- 移动端原生 App。
- Webhook、插件系统、第三方自动化集成。

这些功能可以在架构中预留扩展点，但不能挤占 MVP 主路径。

## 页面列表

### 首页 `/`

目标：让用户快速理解 PolyMind 是多 AI 讨论空间，并直接开始创建。

页面内容：

- 顶部导航：品牌、空间、设置。
- 首屏主操作：新建讨论空间。
- 场景入口：项目头脑风暴、代码评审、论文讨论、商业分析、学习辅导、写作改稿、产品设计。
- 最近空间列表：展示最近的讨论空间和状态。

设计要求：

- 不做空泛营销长页。
- 首屏必须让用户能直接进入产品动作。
- 视觉上体现 AI 思维空间，而不是会议室。

### 空间列表 `/spaces`

目标：管理所有讨论空间。

功能：

- 搜索空间。
- 按更新时间、创建时间、状态筛选。
- 创建新空间。
- 查看空间卡片：主题、参与者数量、最后活动时间、总结状态。

### 新建空间 `/spaces/new`

目标：用低摩擦方式创建讨论空间。

表单字段：

- 讨论主题，必填。
- 讨论目标，选填。
- 背景上下文，选填。
- 讨论模式：头脑风暴、评审、辩论、辅导、写作、分析、自定义。
- 默认轮次上限。

### 空间详情 `/spaces/[spaceId]`

目标：承载完整讨论体验。

区域：

- 顶部空间栏：主题、状态、启动/暂停/结束、生成总结。
- 左侧参与者栏：AI 列表、模型、角色摘要、在线状态、当前发言状态。
- 中心讨论流：消息时间线、流式输出、引用关系、轮次分隔。
- 右侧上下文栏：空间目标、背景、调度设置、总结预览。

核心交互：

- 添加或编辑 AI 参与者。
- 启动讨论。
- 实时查看 AI 发言。
- 暂停或结束讨论。
- 生成和查看总结。

### 参与者编辑 `/spaces/[spaceId]/agents/[agentId]`

也可以实现为 Modal 或 Sheet。

字段：

- 名称。
- 头像或视觉标识。
- Provider。
- 模型。
- API Key 来源。
- 角色设定。
- 背景信息。
- 发言风格。
- 任务定位。
- 温度、最大输出长度等模型参数。

### 设置 `/settings`

第一版仅包含个人本地配置，不做团队设置。

功能：

- 默认 Provider 配置。
- 默认模型。
- OpenAI-compatible Base URL。
- API Key 管理。
- API Key 连通性测试。

## 核心模块

### Space 模块

负责讨论空间的创建、编辑、列表和状态管理。空间是所有讨论数据的聚合根。

核心对象：

- 主题。
- 目标。
- 背景上下文。
- 模式。
- 状态。
- 调度配置。

### Agent 模块

负责 AI 参与者配置。

核心对象：

- 展示信息：名称、颜色、头像标识。
- 模型配置：Provider、模型、Base URL、参数。
- 行为配置：角色、人设、背景、发言风格、任务定位。
- 默认设定：当用户不填写角色时使用通用助手提示词。

### Discussion 模块

负责讨论运行过程。

核心能力：

- 创建讨论运行记录。
- 按轮次选择发言者。
- 将历史消息组织为模型输入。
- 处理流式输出。
- 记录消息状态。
- 支持暂停、继续、结束。

### Summary 模块

负责总结生成。

输出结构：

- 一句话结论。
- 主要共识。
- 关键分歧。
- 支撑理由。
- 风险与不确定性。
- 建议行动。
- 后续追问。

### Provider 模块

负责屏蔽不同 AI API 差异。

第一版必须支持：

- OpenAI。
- OpenAI-compatible。

后续扩展：

- Anthropic。
- Gemini。
- DeepSeek。
- Qwen。

### Settings 模块

负责 API Key、默认 Provider 和模型偏好。

第一版要求：

- API Key 只在服务端使用。
- 数据库存储前必须加密。
- 前端永不展示完整密钥。
- 支持测试 Provider 连通性。

## 开发阶段

### Phase 0：项目脚手架与设计基础

交付：

- Next.js App Router 项目。
- Tailwind CSS、shadcn/ui、Framer Motion、Lucide React。
- 基础 layout、主题变量、全局样式。
- Prisma 和 PostgreSQL 连接。
- 基础 lint、typecheck、格式化脚本。

验收标准：

- `npm run dev` 可启动。
- `npm run lint` 和 `npm run typecheck` 可运行。
- 首页有产品级视觉框架，不是默认模板。

### Phase 1：空间与参与者配置

交付：

- 空间列表。
- 新建空间。
- 空间详情静态框架。
- 添加、编辑、删除 AI 参与者。
- Provider 和模型配置表单。

验收标准：

- 用户能创建空间并添加多个 AI。
- 不填写角色也能保存默认助手配置。
- 表单校验清晰，错误提示友好。

### Phase 2：Provider Adapter 与单 Agent 调用

交付：

- Provider Adapter 抽象。
- OpenAI Adapter。
- OpenAI-compatible Adapter。
- API Key 解密和服务端调用。
- 单个 Agent 流式发言。

验收标准：

- 可在空间内触发一个 AI 流式回复。
- 错误状态可展示并可重试。
- API Key 不进入客户端 bundle。

### Phase 3：多 Agent 讨论调度

交付：

- 讨论运行记录。
- 轮次调度器。
- 多 Agent 依次流式发言。
- 暂停、继续、结束。
- 消息持久化。

验收标准：

- 至少 2 个 AI 可以围绕同一主题连续讨论。
- 每条消息保存发言者、轮次、状态和内容。
- 讨论过程中刷新页面后能恢复当前历史。

### Phase 4：总结与体验打磨

交付：

- 总结生成。
- 总结面板。
- 讨论状态可视化。
- 空状态、加载态、错误态。
- 响应式适配。

验收标准：

- 总结结构稳定，便于复制和继续行动。
- 主要页面在桌面端和移动端无明显布局问题。
- 首屏、空间详情和讨论流达到产品级观感。

## 风险点

- 多 Agent 讨论成本不可控：需要轮次上限、最大 token、停止按钮和预算提示。
- Provider API 差异：必须通过 Adapter 抽象隔离模型调用方式。
- 流式输出与持久化一致性：需要处理连接中断、部分输出、重试和消息状态。
- 提示词堆叠过长：需要上下文压缩策略和消息裁剪策略。
- 用户 API Key 安全：必须服务端调用、加密存储、脱敏展示。
- UI 容易变成聊天模板：空间详情需要突出参与者、轮次、思维过程和总结，而不是只做普通聊天窗口。
- 讨论质量不稳定：需要默认讨论协议，让 AI 学会引用、质疑、补充和收敛。
- 数据模型过早复杂化：第一版保留扩展字段，但不要把团队、计费和插件系统提前做进主流程。

## 后续扩展方向

- Agent 模板库：预设产品经理、架构师、审稿人、投资人、导师等角色。
- 文件上下文：上传文档、代码片段、论文摘要作为讨论材料。
- 讨论模式升级：自由讨论、严格辩论、评审委员会、苏格拉底辅导。
- 多 Provider 对比：同一角色使用不同模型交叉验证。
- 记忆与复盘：为空间沉淀长期上下文和决策记录。
- 协作能力：多人共享空间、评论、权限和邀请。
- 成本控制：用量统计、预算上限和模型推荐。
- 自动报告：将讨论结果导出为 Markdown、PDF 或 Notion 文档。
- 插件与工具调用：后续允许 Agent 调用搜索、代码执行、数据库查询等工具。
